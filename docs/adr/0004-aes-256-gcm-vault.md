# ADR 0004: Use AES-256-GCM for the provider-key vault

- **Status:** Accepted
- **Date:** 2024-01-01
- **Supersedes:** —
- **Superseded by:** —

## Context

We need to store LLM provider API keys (OpenAI, Anthropic, Azure, etc.) so that prompts can be bound to a specific key by reference (`api_key_ref`) rather than embedding the key in the prompt definition. The keys must be encrypted at rest.

The threat model is: an attacker who reads the SQLite database file but does not have the encryption key must not be able to recover the plaintext keys. The encryption must be authenticated (an attacker who tampers with a ciphertext must not be able to read the resulting plaintext).

## Decision

We use AES-256-GCM. The encryption key is a 32-byte (64 hex character) value read from the `PROMPTSHEON_VAULT_KEY` environment variable. A fresh 12-byte random nonce is generated for every encryption; the nonce is prepended to the ciphertext. The output is hex-encoded for storage.

The vault code rejects the all-zero key with a clear error. AES itself would happily accept a zero key, but the resulting key schedule is also all-zero, and ciphertexts are then trivially decryptable by anyone who knows the encoding scheme.

## Options considered

1. **AES-256-GCM (chosen).** Authenticated, NIST-approved, available in `crypto/aes`, hardware-accelerated on modern CPUs.
2. **AES-256-CBC + HMAC.** Two primitives, two keys, two implementations. Weaker on the developer-experience axis; no benefit for us.
3. **ChaCha20-Poly1305.** Equally strong, faster on CPUs without AES-NI. Adds a dependency we do not need.
4. **Hashing only.** Insufficient: the keys must be reversible to be used.

## Consequences

Positive:

- AEAD. Tampering with a ciphertext fails the GCM tag check, returning an error rather than a corrupted plaintext.
- The vault key is a single environment variable. Rotating the key is a deploy operation.
- The implementation is ~80 lines. It is short enough to audit at a glance.

Negative:

- Key rotation is manual. We do not yet support envelope encryption or KMS integration. Re-encrypting all stored keys requires a script.
- The nonce is generated by `crypto/rand` and is unique with overwhelming probability, but we do not enforce a 96-bit nonce uniqueness guarantee at the application layer.
- Loss of `PROMPTSHEON_VAULT_KEY` is unrecoverable. The on-disk ciphertexts are unrecoverable. We document this in [Troubleshooting](../troubleshooting.md).
